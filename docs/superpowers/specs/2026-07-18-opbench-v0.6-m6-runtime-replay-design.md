# OpBench v0.6 M6 Runtime, Replay, and Canary Design

日期：2026-07-18

状态：已批准的设计基线，等待书面规格复核与实施计划

## 1. 目标

M6 在 M1～M5 的 Contracts、Manifest、AgentTaskView、Authoritative Workspace、
Canonical Action Service、AttemptSession、Fresh Evaluator 和 Integrity 之上建立独立的
v0.6 Runtime Spine。它必须冻结 Runtime 语义，证明资源只属于当前 Attempt，执行
Local/Remote Conformance，完成 v0.5 的 17 Baseline、17 Gold、51 Legacy Patch Replay，
并通过标准 Adapter 运行真实 Codex Canary。

M6 不把 v0.5 历史结果重命名为 v0.6 成绩，不扩数据集，也不通过削弱合同、减少
Replay 范围或只使用模拟器来替代真实验收。

## 2. 已确认约束

- 开发继续在本地分支 `codex/opbench-v0.6` 上进行。
- `scripts/run_experiment.py` 未显式选择 v0.6 Runtime 时保持 v0.5 默认行为。
- 先完成离线合同、fixture、Local CPU 和 Replay 门禁，再运行外部 Canary。
- 允许使用用户明确配置的 OpenAI Provider 和 Remote Runtime；禁止主机发现、端口扫描、
  网络探针、广域身份校验和寻找可用机器。
- Remote/CUDA 只使用一个明确配置的目标；不可用时记录可复核的 `Blocked` 证据。
- Task data-plane 网络按 Profile 默认 denied；Controller 到 Provider 的连接不受该规则影响。
- Credential、Remote Host、identity file 和本机绝对路径不得进入仓库或公开 Artifact。
- 资源创建、停止、列举和清理只允许精确的 Attempt-owned identity；禁止广域 `pkill`、
  模糊 container filter 或未绑定 Attempt 的递归清理。
- R-09 真实 Codex CPU Canary 和 R-11 小规模真实批量必须 Passed；代表性 Remote/CUDA
  Canary 在资源确实不可用时可以按 Profile 标为 `Blocked`，不能伪造 Passed。

## 3. 方案选择

### 3.1 采用：独立 v0.6 Runtime Spine

新增 Runtime Contracts、Attempt Resource Ledger、Runtime Backends、Conformance Runner、
Replay Runner 和 v0.6 Orchestrator。底层可以复用现有 Local/Docker/RemoteDocker 命令执行
能力，但所有操作必须经过新的 Profile 和 ownership 层。

这一方案保持 Legacy 默认路径不变，并让 v0.6 的资源、执行、Replay 和 Canary 共享同一
权威数据流。

### 3.2 未采用：原地改造 Legacy EnvironmentManager

直接把新合同塞入现有 `EnvironmentManager` 和 `run_experiment.py` 文件更少，但会继续
耦合 Legacy 与新协议，使默认行为、失败归因和资源 ownership 难以独立证明。

### 3.3 未采用：Fixture/模拟器为主

模拟 backend 适合 TDD 和故障注入，但不能替代 Local 真实执行、17+17+51 Replay、真实
Codex Canary 或资源 cleanup 证据，因此只能作为测试层，不能成为 M6 架构主体。

## 4. 架构边界

### 4.1 Runtime Contracts

`RuntimeProfile` 继续是 Manifest、Session 和 Evaluation 的可比性轴，并新增完整策略：

- `MountPolicy`：逻辑 workspace target、source access、artifact access、root filesystem
  模式；不包含宿主路径。
- `ResourcePolicy`：CPU milli-cores、memory bytes、PID limit、GPU count；不适用的额度显式
  为 null，不依赖 backend 隐含默认值。
- `CleanupPolicy`：`attempt_owned_only` scope、grace、timeout、是否移除 workspace、process
  和 container。
- `hardware` identity：描述公开、可比的 hardware class，不包含 hostname。
- 现有 backend、runtime tier、source loading、platform、image、network 和 timeout 字段继续
  作为 Profile 的精确字段。

Profile 的 canonical identity 对上述任一轴变化敏感。实际 Remote Host 只存在于私有
`RuntimeTargetBinding`，公开侧最多携带其 canonical hash 和声明的 hardware class。

### 4.2 Attempt Resource Ledger

每个资源 identity 由以下字段确定性派生：

```text
attempt_id + retry_index + runtime_profile_hash + resource_type + ordinal
```

资源类型仅包括当前 M6 所需的 `workspace`、`process`、`container` 和
`remote_workspace`。append-only `runtime_resources.jsonl` 记录连续 sequence、前序 hash、
资源 identity、Profile identity 和状态转换：

```text
declared -> created -> released
                   -> cleanup_failed
declared -> create_failed
```

底层 process ID、container name、remote path 等 raw handle 保存在 private runtime
artifact；公开 ledger 只保存逻辑 resource identity 与 raw handle hash。任何缺失、重复、
非法转换、跨 Attempt handle 或无法解释的 active resource 都使 Integrity 失败。

Ledger 不提供全局资源发现 API。恢复和 cleanup 只能读取当前 Attempt 的 descriptor-bound
private lease store，再对精确 handle 操作。

### 4.3 Runtime Backend

统一 backend 接口负责：

```text
prepare(profile, attempt_context) -> RuntimeLease
run(lease, command, cwd, timeout_ms) -> RuntimeCommandResult
collect(lease) -> RuntimeEvidence
cleanup(lease) -> RuntimeCleanupResult
```

实现分为：

- Local Process：真实本地进程 fixture 和 Local CPU Profile；
- Local Docker：精确 container name/label 和 workspace mount；
- Remote Docker：明确 `RuntimeTargetBinding`、精确 remote workspace/container handle；
- Scripted Backend：仅用于故障注入和跨 backend Conformance 单元测试。

现有 Executor 可作为命令执行细节复用，但不得绕过 Resource Ledger。Remote backend 不实现
ping、scan、host discovery、port check 或枚举远端 container/process。

### 4.4 Conformance Runner

Conformance Runner 对同一冻结 Source、TaskView、Capability/Budget Policy 和 Canonical
Action Sequence 执行：

- CLI 与 MCP transport；
- Local Process 与可用的 Docker/Remote backend；
- patch freeze、fresh evaluation、cleanup 和 integrity。

比较结果使用规范化 `ConformanceSnapshot`。Action Observation、Budget、Patch、Session
Result、Evaluation Result 和 Cleanup 状态必须相同；timestamp、duration、private host、
container raw handle 和本机路径不参与等价比较。忽略字段集合由版本化 contract 固定，
不能由调用者任意传入。

### 4.5 Replay Runner

Replay Runner 只读取冻结的 v0.5 Dataset 和历史 Artifact：

- 17 个 baseline case 来自正式 Dataset 与已验证 baseline evidence；
- 17 个 gold case 来自每个 Task 的冻结 gold patch；
- 51 个 Legacy Final Patch 来自四个正式 patch root：Legacy CPU 30、Legacy CUDA 3、
  Precision CPU 9、Precision GPU 9。

每个 `ReplayCase` 绑定 Task、Source、Environment、Runtime Profile、patch identity、历史
expected outcome 和 artifact provenance。重复、缺失、未知 Task、attempt 重号、patch path
逃逸或历史 expected outcome 不一致都在执行前失败。

Replay 重新调用 v0.6 Fresh Evaluator，并输出逐项 expected/observed/difference attribution。
它不创建 Agent Session，不写入 v0.6 Agent Cohort，也不改变 v0.5 results/summary。

### 4.6 v0.6 Orchestrator

只有显式 `--runtime-protocol v1 --runtime-profile <id>` 才从
`scripts/run_experiment.py` 分派到 v0.6 Orchestrator。默认参数继续运行 Legacy 路径。

v0.6 Orchestrator 依次调用：RunManifest validation、Runtime preparation、Workspace、
AttemptSession、标准 Agent Adapter、Patch Freeze、Fresh Evaluator、Runtime cleanup、
Integrity 和 Summary rebuild。Conformance、Replay 和 Canary 复用这些组件，不复制另一套
运行逻辑。

真实 Codex 只接收 M3 已定义的 `AdapterContext` 与 JSON-only action client，不得到 Full
Task、Workspace、Evaluator、Remote binding 或资源 raw handle，也不增加 Canary 专用能力。

## 5. 数据流

一个 v0.6 Attempt 的固定顺序是：

1. 校验 RunManifest、Task、Agent 和完整 RuntimeProfile；无效输入不创建资源。
2. 创建 retry-specific Runtime Resource Ledger，并声明预期资源 identity。
3. backend 创建并登记 workspace、process/container 和可选 remote workspace。
4. 启动 AttemptSession，通过标准 Adapter 执行 Canonical Actions。
5. Session terminal 后 Freeze 唯一 patch，并在同一 Profile 语义下 Fresh Evaluate。
6. `finally` 路径对 ledger 中的精确 lease 执行 cleanup，记录每项状态。
7. Integrity 同时校验 Attempt Artifact、Resource Ledger、private lease hash 和 cleanup report。
8. 只有 cleanup 和 Integrity 均满足合同，Attempt 才可进入最终 Summary。

Conformance 调用同一数据流两次或多次后比较规范化 snapshot。Replay 从 `ReplayCase` 直接
进入 Runtime preparation/Fresh Evaluation，不经过 Agent 阶段。Canary 通过标准 Adapter
进入完整数据流。

## 6. 失败归因

| 阶段 | 稳定归因 |
| --- | --- |
| Profile/Manifest validation | 运行前 ContractError，不创建 Attempt 资源 |
| Source/workspace materialization | `workspace_error` |
| backend/container/process start | `platform_error` |
| Agent Action runtime | `runtime_error` |
| Provider invocation | `provider_error` |
| Fresh Evaluator backend | `evaluation_error` |
| Attempt deadline | `timeout` |
| Runtime cleanup | 原 Session/Evaluation 保留，Integrity `cleanup_failed` |
| 明确 Remote/CUDA target 不可用 | Canary/Replay `Blocked`，不进入 Agent 分数 |

cleanup failure 不得覆盖原始 Agent/Evaluation 归因，也不得静默视为通过。无法证明资源已经
released 时，Integrity 必须 fail closed。

## 7. 网络与私有数据边界

- Runtime 代码不枚举网络、主机、端口、container 或全局进程。
- Remote backend 只读取用户明确指定的 private target binding。
- 不新增 ping、scan、host-key 探测、地址发现或寻找可用机器逻辑。
- 不对仓库外未知路径、未知 process 或未知 container 执行 cleanup。
- Provider 网络仅由既有 Codex CLI 路径使用，Runtime 不自行调用 Provider API。
- task data-plane network 默认 `denied`；需要变化时必须改变 RuntimeProfile identity。
- hostname、user、identity file、Credential、本机绝对路径和 raw handle 只存在于 private
  control-plane artifact；public scanner 必须拒绝泄漏。

## 8. CLI 与 Artifact

### 8.1 CLI

- `scripts/run_experiment.py`：默认 Legacy；显式 v1 参数分派到新 Orchestrator。
- `scripts/run_runtime_conformance.py`：离线/显式 target Conformance。
- `scripts/run_legacy_replay.py`：构建并执行冻结 Replay Matrix。
- `scripts/verify_runtime_resources.py`：只读校验一个 run 的 Resource Ledger/cleanup。

外部 Canary 默认关闭；命令必须显式给出 Profile 和 private target config。命令不尝试寻找
替代 target。

### 8.2 Artifact layout

```text
runs/<cohort>/
  attempts/<attempt-id>/retries/retry-NNNN/
    runtime_resources.jsonl
    private_runtime_resources.json
    runtime_cleanup.json
    runtime_conformance.json
  replay/
    replay_manifest.json
    replay_results.jsonl
    replay_differences.json
    replay_summary.json
```

`private_runtime_resources.json` 不进入公开包。公开文件使用 canonical JSON、固定 schema
version、描述符绑定写入和现有敏感值扫描。

## 9. 测试与验证

### 9.1 Contract/Schema

- Local CPU、Remote CPU、CUDA Overlay、CUDA Kernel Build 与 CPU Compile Profile round-trip；
- Mount/Resource/Cleanup Policy 的非法字段、未知版本、角色错配和不可能组合；
- Profile 任一有效轴变化都会改变 comparability/attempt identity；
- Legacy migration 产生完整、确定性的新 Profile，同时保持旧 CLI 默认路径。

### 9.2 Resource ownership

- 并发 declared/created/released 收敛；
- 重复 release 幂等，冲突 release fail closed；
- 跨 Attempt raw handle、目录替换、symlink、尾部损坏和重算 hash 篡改失败；
- create/start/command timeout/cleanup failure 均有确定状态；
- backend 只能接收当前 ledger 创建的 lease；
- 校验代码不调用全局枚举或无归属 cleanup。

### 9.3 Backend/Conformance

- Local Process 使用真实临时 Git fixture 和真实子进程，不用 mock 替代主路径；
- Docker/Remote 使用命令构造、ownership 和故障注入 fixture；
- 可用资源上的显式 Canary 提供正常集成证据；
- CLI/MCP、Local/Remote snapshot 对规范化字段等价；
- 不可比字段只能来自固定 allowlist。

### 9.4 Replay

- discovery 精确得到 17 Baseline、17 Gold、51 Legacy Patch；
- 四个 Legacy patch root 的数量分别为 30/3/9/9；
- 缺失、重复、未知 Task、attempt 重号、path escape、expected outcome 改写均失败；
- 每个 case 输出独立结果和差异归因；
- Legacy Summary 与 v0.6 Replay Summary 分离；
- Replay 不创建 Agent 成绩。

### 9.5 Canary/compatibility

- 至少一个真实 Codex CPU Attempt 完成 read/edit/test/finish、Fresh Evaluation、cleanup、
  Integrity 和 Summary；
- 小规模真实 Codex batch 验证 resume、retry、summary 和 attribution；
- Remote CPU/CUDA Overlay/CUDA Kernel 按明确 target 执行或记录可复核 Blocked；
- v0.5 CLI、results.jsonl、summary 和默认行为回归通过。

### 9.6 固定门禁

- M6 focused tests；
- 全部 runtime tests；
- 全量 tests；
- 17-task Dataset validation；
- RunManifest/Schema validation；
- tracked JSON validation；
- compileall；
- `git diff --check`；
- 独立审查 Critical/Important 为 0。

| Acceptance | 设计证据 |

## 11. 退出条件

M6 仅在以下条件同时成立时完成并提交里程碑：

1. R-01～R-12 每项达到其允许的声明状态；R-09 和 R-11 必须 Passed。
2. 17+17+51 Replay 全部有逐项记录；无法执行项有 Task/Environment/Protocol 级解释。
3. 至少一个真实 Codex CPU Attempt 通过标准 Adapter 和 Integrity。
4. focused/runtime/full、Dataset、Schema、tracked JSON、compileall、diff check 全部通过。
5. 独立复审 Critical/Important 为 0。
6. 没有运行网络探针、主机发现或无归属资源清理。

## 12. 非目标

- 不新增 Task、Dataset slice、排行榜、Web UI、数据库或调度服务。
- 不重写 v0.5 历史结果，不把 Replay 当作新 Agent 分数。
- 不新增 Agent 专用 shortcut 或绕开 Canonical Action Service 的 Canary 路径。
- 不实现多主机发现、资源池搜索或广域 cleanup。
- 不因 Remote/CUDA 暂不可用而削弱 Local、Replay、Integrity 或真实 Codex CPU 要求。
