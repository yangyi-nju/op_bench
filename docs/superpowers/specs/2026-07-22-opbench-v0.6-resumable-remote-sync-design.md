# OPBench v0.6 Resumable Remote Sync Design

## Context

The v0.6 CUDA Kernel experiment freezes the exact PyTorch revision together
with 73 recursive submodules. The resulting controller workspace is about
1.7 GB. Two consecutive formal attempts created their exact remote workspace,
then failed during the initial `rsync` after 477 seconds with
`session_platform_error`; neither attempt launched Codex. The configured
filesystem had about 1.1 TB free, so capacity was not the limiting resource.

The current backend performs one `rsync` call. Any transport interruption
causes the attempt-owned remote workspace to be deleted, so the next retry
must transfer the entire frozen tree again.

A controlled reproduction later completed the exact 1.7 GB rsync in 398
seconds with exit code 0, while the immediately following new SSH connection
returned 255. This establishes a second transient boundary at the exact ccache
seed copy that follows a long transfer.

## Decision

Make remote workspace synchronization resumable inside one Runtime attempt:

- add `--partial` to the exact `rsync` command so an interrupted file remains
  available to the next invocation;
- execute the same command against the same attempt-owned remote workspace at
  most three times;
- return immediately after the first successful invocation;
- after three failures, preserve the existing
  `remote_source_sync_failed` reason and existing exact-resource cleanup;
- use the same helper for initial preparation and later workspace re-syncs.
- make the ccache seed copy repeatable by copying `seed/.` into the
  attempt-owned `.ccache` destination, and retry that exact command at most
  three times;
- preserve `remote_ccache_seed_copy_failed` when all seed-copy attempts fail.

No sleep, backoff, target discovery, alternate host, or broad cleanup is
introduced. The helper does not allocate a new resource and does not change
the manifest, Runtime Profile, source fingerprint, resource identity, or
evaluation contract.

## Alternatives Rejected

An immutable remote source seed would make startup faster, but it needs a new
target configuration field, seed identity verification, lifecycle rules, and
cache invalidation semantics. Splitting the root and submodule transfers would
duplicate source-materialization structure in the transport layer. Both are
larger changes than a bounded continuation of the existing exact transfer.

## Verification

One deterministic fake runner fails the first two `rsync` invocations and
succeeds on the third. A second fails the first two exact ccache-copy SSH
commands and succeeds on the third. The tests require identical destinations,
repeatable copy operands, no premature remote cleanup, a usable lease, and
complete cleanup afterward. The existing permanent sync/cleanup failure test
continues to prove the retry bound and failure cleanup behavior.
