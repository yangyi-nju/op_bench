# OpBench v0.6 Real MCP Full-Agent Experiment Design

日期：2026-07-20

状态：已批准的设计基线，等待书面规格复核与实施计划

## 1. 目标

在已经完成的 OpBench v0.6 平台上补齐真实 Codex 到 Canonical Action Service 的
attempt-scoped MCP stdio 接入，并使用该接入对冻结的 `pytorch_v0.5` 17-task Dataset
执行三重复全量实验，共 51 个真实 Agent Attempt。

这次工作必须同时证明两件事：

1. 真实 Codex 可以只通过 MCP tools 完成 read/edit/test/diff/finish，且 MCP 与现有 CLI
   compatibility transport 共享相同 Action、Budget、Trace、Patch Freeze 和 Fresh
   Evaluation 语义。
2. v0.6 的四类正式 Remote Runtime Profile 可以承载完整 Agent cohort，并生成可续跑、
   可归因、可重建、可审计的 Trace、Integrity、资源清理和实验汇总证据。

实验结果允许是 `resolved`、`f2p_failed`、`no_patch` 或 `timeout`。本设计不要求 Agent
解决全部 Task，也不为了提高分数重跑合法失败或改写原始结果。

## 2. 当前基线与已确认约束

- 开发和实验继续在本地分支 `codex/opbench-v0.6` 上进行。
- 设计基线 HEAD 为 `349ce7fd9abab9db2c147adbbbe2fc8c8e9802b9`；实现完成后以新的
  MCP implementation commit 作为正式实验 platform identity。
- Dataset 固定为 `datasets/pytorch_v0.5/dataset.json` 的 17 个 verified Task。
- Agent 固定为真实 Codex CLI，三重复，共 51 个独立 Attempt。
- 当前本机 Codex CLI 为 `codex-cli 0.145.0-alpha.18`。正式实验在 canary 前重新读取并
  冻结实际版本；实现和 canary 期间 CLI 版本变化必须导致身份变化并拒绝错误 resume。
- 正式实验必须显式传入非空 Codex model ID，不再使用含糊的
  `codex-cli-configured` 作为实验模型身份。
- Codex 认证继续使用现有 `CODEX_HOME`；MCP 配置只通过本次 `codex exec` 的 `-c`
  overrides 注入，不修改用户全局 `config.toml`。
- Codex 使用 `--ignore-user-config` 隔离其他 MCP server、工具配置和模型默认值；认证数据
  仍由 Codex CLI 自己读取。所选 model 通过 `--model` 显式传入。
- Runtime 只使用 `configs/remote_hosts.json` 中已经配置的精确 target。禁止 ping、端口或
  服务扫描、主机发现、替代目标搜索、宽泛进程/container 枚举和无归属 cleanup。
- Task data plane 继续按 Runtime Profile 使用 `network_policy=denied`；Controller 到 Codex
  Provider 的正常连接不属于 Task data plane。
- 四个 Runtime Profile 分开形成四个 Cohort；不同 Comparability Key 只在实验索引中并列，
  不合并成一个虚假 Cohort。
- v1 runner 保持串行。CUDA Profile 不并发，避免争用唯一 GPU allocation。
- private evaluation、private runtime handles、target binding、Credential 和本机绝对路径不
  进入公开 Artifact 或提交。

## 3. 方案选择

### 3.1 采用：真实 MCP 接入后执行 51-Attempt 全量实验

先增加 attempt-scoped stdio MCP Server 和独立 `codex_mcp_canonical` Adapter，通过离线
测试、Local CPU real-Codex canary 和精确 Remote CPU canary 后，再执行四个三重复正式
Cohort。

这一方案直接验证 v0.6 新增的 MCP、Trace、Integrity、Runtime 和 Fresh Evaluation，且不
需要重写 Canonical Action Service。

### 3.2 未采用：直接用现有 CLI compatibility Adapter 跑 51 次

现有 `codex_canonical` 已可运行真实 Codex，但它通过 `opbench_action.py` 命令行和
ProcessActionExchange 工作，不能证明真实 Agent 的 MCP handshake、tool discovery 和
tool-call lifecycle。它保留为兼容和回归路径，不作为本次正式实验 Adapter。

### 3.3 未采用：CLI 与 MCP 各跑完整 51 次

双全量可以比较 transport，但会增加到 102 个真实 Agent Attempt，成本和耗时过高。CLI/MCP
语义等价由 deterministic conformance 证明；正式质量实验只使用 MCP。

## 4. 架构

### 4.1 固定数据流

```text
Codex CLI
  | stdio JSON-RPC / MCP tools
  v
Attempt-scoped MCP Server
  | protected process action client
  v
ProcessActionExchange
  | canonical descriptor-bound JSON IPC
  v
CanonicalActionService
  |
  +-- Authoritative Workspace
  +-- AttemptSession / Budget
  +-- EventJournal / Trace
  +-- FrozenPatch
  +-- Fresh Evaluation
```

MCP Server 和 Codex 都只能看到 attempt-private scratch、public AgentTaskView 和 Canonical
Action observations。它们不得获得 FullTaskSpec、hidden tests、Workspace 对象、Evaluator、
Remote binding 或 raw resource handles。

### 4.2 MCP protocol boundary

新增 stdio MCP server，只实现真实 Codex 所需的协议面：

- JSON-RPC `initialize`；
- `notifications/initialized`；
- `ping`；
- `tools/list`；
- `tools/call`。

Server 支持 MCP protocol versions `2024-11-05`、`2025-03-26` 和 `2025-06-18`。当 client
提出其他版本时，server 返回其最高支持版本 `2025-06-18`，由 client 决定是否接受
downgrade。stdio 每行只允许一个 UTF-8 JSON object，单条消息上限 1 MiB；stdout 只写
JSON-RPC，诊断只写 stderr。

`tools/list` 只公布以下九个 versioned tools：

- `workspace_list`
- `workspace_search`
- `workspace_read`
- `workspace_write`
- `workspace_apply_patch`
- `command_run`
- `test_run`
- `vcs_diff`
- `session_finish`

Tool input schema 来自一个 canonical registry，MCP `tools/list`、prompt 和测试共用它，不
维护三份手写 schema。Agent 只提交 action arguments；session ID、action ID、client
sequence 和 deadline 由 attempt-bound process client 注入。

`tools/call` 的正常返回同时提供 MCP text content 和 `structuredContent`，其内容是完整、
公开、canonical `ActionObservation`。Action 的 `ok=false` 仍是成功完成的 MCP tool call，
不是 JSON-RPC protocol error。

### 4.3 Process isolation and IPC

复用现有 `ProcessActionExchange`：

1. Controller 创建 mode `0700` 的 exchange root、request/response directories 和
   descriptor-bound `opbench_action.py`。
2. Adapter 把一个只依赖 Python 标准库的 MCP server launcher 复制到 attempt scratch，权限
   固定为 `0700`。
3. Codex CLI 通过 invocation-local `mcp_servers.opbench` 配置启动该 server。server 只接收
   受保护 action client 的精确路径，不接收 target、workspace 或 hidden asset 路径。
4. 每个 MCP tool call 由 server 调用 action client；action client 生成 canonical request，
   ProcessActionExchange 转交当前 Attempt 的 `AdapterActionClient`。
5. Controller 中唯一 `CanonicalActionService` 执行请求并写入 EventJournal；response 原路返回
   MCP client。

Server 不复制 Action validation、Budget、Workspace、Patch 或 Evaluation 规则。

### 4.4 Codex MCP Adapter

新增独立 Adapter ID `codex_mcp_canonical`。它与 `codex_canonical` 共用 provider status 和
terminal classification，但必须拥有独立 Agent identity 和 Comparability Key。

Adapter 启动 Codex 时固定：

- `codex exec --ephemeral --ignore-user-config`；
- 显式 `--model <exact-model-id>`；
- `--cd <attempt-scratch>`；
- `--skip-git-repo-check`；
- `--sandbox workspace-write`，可写范围只有 attempt scratch；
- invocation-local `mcp_servers.opbench.command/args/env`；
- 不加载其他 MCP server。

Prompt 只说明 Task、九个 MCP tools、最小修改原则、测试和 Finish 要求，不再要求调用
`./opbench_action.py`。Codex 的普通 shell 即使被调用也只能看到 scratch，目标 repository
仍不可直接访问。

Codex、MCP server 和其精确 descendants 属于同一个新进程组。正常完成、timeout、Provider
failure、KeyboardInterrupt 和 Adapter exception 都通过记录的 process-group identity 收敛；
禁止 `pkill`、进程搜索或按名称宽泛清理。

### 4.5 CLI and identities

`scripts/run_experiment.py` 扩展为接受：

- `--agent codex_mcp_canonical`；
- `--codex-model <exact-model-id>`，对真实 MCP Adapter 必填；
- 现有 `--enable-external-canary` 继续作为显式 Provider opt-in；
- 现有 v1 Dataset、Profile、target、repeat 和 resume 参数保持不变。

Agent identity 至少绑定 adapter ID、exact model ID、Codex CLI version、action protocol、MCP
protocol capability set 和 prompt version。上述任一字段变化都会改变 Agent/Cohort/Attempt
identity，旧 output root 必须拒绝混入新配置。

## 5. Trace and artifact semantics

每个 accepted MCP `tools/call` 必须对应：

1. 一个 canonical ActionRequest；
2. 一个 canonical ActionObservation；
3. EventJournal 中连续、hash-chained 的 request/observation batch；
4. 一次确定的 Budget delta；
5. 可选的一次 Workspace mutation state transition。

MCP initialize、tool discovery、JSON-RPC error 和 server lifecycle 写入独立 public adapter
trace summary，不伪装成 Action event。Provider stdout/stderr、MCP private argv/environment 和
Credential 不进入 public trace。

Attempt public artifacts增加 MCP adapter metadata：

- Adapter ID；
- exact model ID；
- Codex CLI version；
- negotiated MCP protocol version；
- MCP initialize/tools-list/tool-call counts；
- protocol error count；
- server terminal status。

这些字段进入 Integrity 引用图和 deterministic rebuild。私有 MCP command、absolute paths 和
process handles 只保存在 controller-private evidence。

## 6. Failure attribution and resume

| Failure | Attribution | Resume |
| --- | --- | --- |
| Manifest/Profile/model/CLI identity invalid | pre-run contract error | 修正输入后使用新 root |
| Codex executable missing/version mismatch | infrastructure invalid | 允许精确 retry |
| Provider auth/rate-limit/quota/unavailable | infrastructure invalid/provider error | 允许精确 retry |
| MCP server start/initialize/tools-list failure | infrastructure invalid/MCP error | 允许精确 retry |
| malformed/oversized JSON-RPC or server crash | infrastructure invalid/MCP error | 允许精确 retry |
| Action Observation `ok=false` | 正常 Agent trace fact | 不自动 retry |
| Agent invalid tool arguments/repeated action error | 正常 Agent behavior | 不自动 retry |
| Agent timeout/missing finish/no patch | Agent terminal + truthful evaluation | 不自动 retry |
| Runtime prepare/evaluation backend failure | infrastructure invalid | 允许精确 retry |
| MCP/Codex/Runtime cleanup failure | fail closed / cleanup_failed | 允许精确 retry |
| KeyboardInterrupt | interrupted, exact cleanup | 相同命令继续缺失 Attempt |

retry 只创建新的 retry index，不覆盖原始失败记录。已经有效结束的 Attempt 即使是
`f2p_failed`、`no_patch` 或 `timeout`，相同命令也必须 skip。

## 7. Testing and gates

### 7.1 Unit and contract tests

- initialize/version negotiation、initialized notification 和 ping；
- tools/list 的九个名称、schema、顺序和 canonical bytes；
- tools/call 的 text/structured response；
- unknown method/tool、invalid params、invalid JSON、oversized message 和 stdout contamination；
- Action `ok=false` 不被错误提升为 protocol failure；
- exact model 和 CLI version 进入 Agent/Comparability identities；
- 非 MCP Adapter 使用 `--codex-model` 时 fail closed；
- MCP Adapter 缺 model、缺 external opt-in 或混入多个 Agent 时在资源创建前失败。

### 7.2 Transport and process integration

- 相同 scripted sequence 经 CLI/MCP 得到等价 Observation、Budget、Patch 和 Event；
- fake Codex 完成 MCP initialize/list/call/read/edit/test/diff/finish；
- fake server crash、Codex timeout、Provider error、duplicate finish 和 malformed response；
- Codex/MCP 精确 process-group cleanup，不调用进程枚举或宽泛 kill；
- exchange root symlink/descriptor replacement、message tampering 和 sequence corruption fail closed；
- resume 只重跑 infrastructure-invalid 或缺失 Attempt；
- MCP metadata、Trace 和 Summary deterministic rebuild byte-identical。

### 7.3 Fixed offline gate

- MCP focused tests；
- 全部 runtime tests；
- 全量 unit test suite；
- 17-task Dataset validation；
- Runtime Profile/RunManifest contract validation；
- tracked JSON parsing；
- `python -m compileall -q src scripts tests`；
- `git diff --check`；
- privacy scan；
- 独立 review 的 Critical/Important 为 0。

## 8. Canary sequence

外部执行仅在 offline gate 全部通过后开始。

### 8.1 Real Local CPU MCP canary

使用已有 M6 local fixture 和真实 Codex MCP Adapter，要求：

- MCP initialize、tools/list 成功；
- 至少完成 read、edit/apply、registered test、diff 和 finish；
- Action/Observation Trace pairing 和 hash chain 完整；
- Fresh Evaluation、14/14 Integrity、resource ownership 和 cleanup 通过。

该 canary 是 Adapter/transport 证据，不发布 Agent 分数。

### 8.2 Exact Remote CPU MCP canary

使用正式 Dataset 中 `pytorch__149693__lazylinear_init`、
`remote-cpu-pytorch-2.6-py311-v1` 和唯一 private target。要求 Attempt 有合法终态、Trace 与
Evaluation 完整、14/14 Integrity、workspace/container/remote workspace 精确释放。

不要求 canary `resolved`；Agent 的合法失败仍可证明端到端平台路径。若出现 infrastructure
failure，只按 exact recorded handle 清理并使用 retry 语义，不搜索其他目标。

## 9. Full experiment matrix

正式实验使用四个不可变 output roots：

```text
runs/v0.6_mcp_full_20260720_remote_cpu
runs/v0.6_mcp_full_20260720_remote_cpu_compile
runs/v0.6_mcp_full_20260720_cuda_overlay
runs/v0.6_mcp_full_20260720_cuda_kernel
```

| Cohort | Runtime Profile | Tasks | Repeat | Attempts |
| --- | --- | ---: | ---: | ---: |
| Remote CPU | `remote-cpu-pytorch-2.6-py311-v1` | 12 | 3 | 36 |
| Remote CPU Compile | `remote-cpu-compile-pytorch-2.6-py311-v1` | 1 | 3 | 3 |
| CUDA Overlay | `remote-cuda-overlay-pytorch-2.6-cu124-v1` | 2 | 3 | 6 |
| CUDA Kernel | `remote-cuda-kernel-pytorch-2.6-cu124-v1` | 2 | 3 | 6 |
| Total | four distinct Comparability Keys | 17 | 3 | 51 |

执行顺序固定为 Remote CPU、Remote CPU Compile、CUDA Overlay、CUDA Kernel。每个 Cohort
完成后立即运行 resource verifier、Integrity/rebuild、JSON/privacy 和 summary checks，再开始
下一个 Cohort。中断后重复该 Cohort 的完全相同命令。

## 10. Experiment acceptance

正式实验完成必须同时满足：

1. 四个 manifests 的 expected attempts 分别为 36、3、6、6，总计 51。
2. 51 个 logical Attempt 全部有 selected result；没有 missing、Blocked 或未处理的
   infrastructure-invalid Attempt。
3. 每个完整 controller-private root 的 14 项 Integrity checks 全部 Passed。
4. 每个 Attempt 的 runtime ownership 和 cleanup 可验证；每个 Cohort 的 resource verifier
   Passed。
5. public Results/Summary 可从冻结 Artifact 确定性重建且 bytes 一致。
6. MCP/Action Trace 的 hash chain、pairing、sequence、budget、Finish 和 adapter metadata 完整。
7. 四个 Cohort 的 exact model、CLI version、Adapter、Dataset、Profile、protocol 和 scoring
   identities 冻结且可复核。
8. 所有合法 Agent outcome 原样保留，不把失败改为基础设施错误，不选择性重跑低分结果。
9. private files 从 public evidence 排除；privacy scan 无 host、user、identity file、Credential、
   raw handle 或本机绝对路径泄漏。
10. 独立最终 review 的 Critical/Important 为 0。

## 11. Final analysis and report

新增只读实验索引/汇总步骤，读取四个 Cohort 而不修改原 Artifact。报告至少包含：

- 每个 Task、repeat 和 Profile 的 terminal、evaluation outcome 和 resolution；
- resolved/f2p_failed/no_patch/timeout 分布及合法 denominator；
- 每个 Action 名称的 call/error count、duration 和 Budget delta；
- read/edit/test/diff/finish coverage；
- Patch bytes/changed files 分布；
- Provider、MCP、Runtime、Agent 四类失败归因和 retry 记录；
- Trace event count、hash-chain/Action pairing/sequence/Finish 完整性；
- CLI 历史实验与本次 MCP 实验的描述性对照。

跨 Profile 只汇总描述性 totals；不会把四个 Comparability Key 伪装成一个 Cohort，也不会在
没有统计设计的情况下宣称 MCP 提高或降低模型能力。

## 12. Public/private evidence boundary

完整 controller-private roots 用于 14/14 Integrity、resource ownership 和 cleanup 验证。
Git 中只发布：

- public RunManifest、Attempt/Result/Summary；
- public Event/Trace、Patch、Evaluation、Integrity 和 cleanup attestations；
- 四 Cohort experiment index 和最终分析；
- canary 的公开子集。

以下文件和值不发布：

- `private_evaluation.json`；
- `private_runtime_resources.json`；
- Provider raw output、Credential 和 auth state；
- MCP private command/env；
- target host/user/identity file/remote root；
- raw PID、process group、container name 和 remote workspace path；
- controller absolute paths。

公开文档必须明确：14/14 是在 redaction 前的完整 controller-private root 上验证；检出的
public subset 不能独立重跑需要 private handles 的完整 Integrity graph。

## 13. Non-goals

- 不新增或修改 Dataset Task。
- 不执行 CLI/MCP 双全量 A/B 实验。
- 不增加 Web UI、数据库、分布式调度器或长期 MCP daemon。
- 不提供 network MCP server；只支持 attempt-scoped stdio。
- 不让 MCP Server 直接持有 Workspace、Evaluator 或 Runtime backend。
- 不改变 Legacy `codex_action_bridge` 或默认 Legacy CLI 行为。
- 不把 transport conformance 当作模型质量因果结论。
- 不推送、打 tag 或发布排行榜，除非后续得到明确指令。

## 14. Exit condition

本工作在以下条件全部满足后结束：

1. MCP implementation、offline gate 和两级 real-Codex canary 通过。
2. 四个正式 Cohort 完成 51 个 logical Attempt，无未处理基础设施缺口。
3. Trace、Integrity、resource cleanup、deterministic rebuild、privacy 和独立 review 通过。
4. 最终实验报告准确区分平台证据、Agent 结果、Profile 可比性和非声明。
5. 全程未执行网络探针、扫描、目标发现、资源枚举或宽泛 cleanup。
