# OpBench v0.6 Revision-Only Source Materialization Design

Date: 2026-07-18

Status: Implemented with the identity-preserving refinement recorded below.

## 1. Context

The first resumed exact-Remote CPU canary reached the configured target, created
its remote workspace and container, and released every Attempt-owned resource.
It then produced `session_workspace_error` before launching the Adapter.

The failure was reproduced locally at the source boundary. The registered source
repository contained an ignored `android/.gradle/` directory. The Runtime backend
used `shutil.copytree` on the whole cache working tree, so ignored and untracked
cache state entered the Attempt workspace. `AuthoritativeWorkspace.open`
correctly rejected that workspace because its cleanliness check includes ignored
files. The tracked `HEAD` tree still matched the Task's frozen base revision, so
this was cache pollution rather than a Task source change.

This exposes a broader correctness defect: a Runtime Attempt currently consumes
the cache working tree while the comparable contract freezes a Git revision.
Ignored files, untracked files, and tracked working-tree edits can therefore
change execution without changing the frozen RunManifest.

## 2. Goals

1. Materialize every executable Runtime workspace from the exact frozen Git
   revision, never from mutable cache working-tree state.
2. Preserve a clean, standalone Git repository because
   `AuthoritativeWorkspace` requires Git status, diff, and patch-freeze
   operations.
3. Apply the same source-purity invariant to Local Process, Docker, and Remote
   Docker backends.
4. Leave the shared source cache byte-for-byte untouched: no checkout, clean,
   reset, worktree registration, or removal of ignored files.
5. Preserve Attempt-only resource ownership, exact cleanup, stable error
   attribution, resume behavior, and public/private artifact separation.
6. Keep network behavior unchanged. The fix must not add probing, discovery,
   scanning, fallback targets, or extra Remote operations.

## 3. Non-goals

- Changing Dataset, Task, Source, Environment, Runtime Profile, scoring, or
  historical v0.5 identities.
- Repairing or deleting source-cache contents.
- Fetching or initializing submodules from a remote, or materializing submodule
  worktrees for Profiles that do not require an in-place source build.
- Replacing the existing source registry or introducing a new artifact store.
- Treating deterministic Scripted-Remote evidence as a substitute for exact
  Remote CPU/CUDA execution.

## 4. Considered approaches

### A. Exact revision archive into an independent snapshot repository

Stream `git archive <revision>` into a new Attempt-owned directory, validate
every archive entry, rebuild the exact tree (including Gitlink index entries),
then install the original commit object as the shallow snapshot `HEAD`. This
excludes all non-versioned cache state, does not mutate the source repository,
and preserves commands such as `git archive HEAD` that participate in source
identity verification.

This is the selected approach. The implementation originally followed the
deterministic-root-commit wording below, but exact replay showed that replacing
`HEAD` broke the frozen source archive identity even when the tree was equal.
Preserving the original commit object is therefore a required correctness
refinement, not a change to the public Source contract. It also supports an
unreachable-but-present commit object in a shallow snapshot repository.

### B. Local clone followed by detached checkout

A clone naturally excludes ignored files and produces a Git repository, but an
original base commit may be present only as an unadvertised object in the
snapshot cache. A normal clone may therefore omit the required revision. It also
copies more Git metadata than an Attempt needs.

### C. Temporary cache cleanup followed by the existing copy

Moving or deleting the ignored Gradle directory would unblock one run but would
not prevent later ignored, untracked, or modified files from changing a frozen
Attempt. It also mutates shared cache state and masks the contract violation.

## 5. Architecture

### 5.1 Private runtime context

`RuntimeAttemptContext` will carry both:

- `frozen_source_directory`: the exact local Git object source; and
- `frozen_source_revision`: the commit selected by the frozen Task/Source
  binding.

The orchestrator and legacy replay path obtain the revision from
`LocalGitSource.revision`. Conformance fixtures explicitly use their fixture
commit. The revision is private execution input; the existing public Source
identity remains the comparable contract, so no public schema or Cohort identity
changes.

### 5.2 Revision materializer

A single backend helper will materialize the source for all executable backend
types:

1. Validate that the source directory is a real local Git repository and that
   `<revision>^{commit}` resolves without consulting a remote.
2. Create the exact Attempt workspace path and no other destination.
3. Stream `git archive --format=tar <revision>` from the local object database.
4. Accept only relative, normalized entries. Reject absolute paths, `..`, device
   nodes, FIFOs, sockets, hard links, or symlinks whose lexical target escapes
   the workspace. Preserve versioned regular files, directories, executable
   modes, and safe relative symlinks.
5. Initialize a standalone Git repository in the extracted directory and add
   the complete extracted tree with ignore rules bypassed. Because `git archive`
   represents Gitlinks as directories rather than index entries, restore every
   `160000` entry directly from the frozen source tree without initializing its
   submodule.
6. Require the rebuilt tree object hash to equal `<revision>^{tree}`. Copy the
   original commit object bytes into the snapshot object database, mark the
   commit shallow when it has parents, and point `HEAD` at that exact commit.
7. Require `HEAD == <revision>`, require `git archive HEAD` to retain the frozen
   source identity, and require a clean status (including ignored and untracked
   files) before publishing the workspace resource as `created`.
8. For `inplace_build` only, recursively read each populated local submodule
   object database at the exact Gitlink commit and safely archive its worktree
   into the Attempt workspace. Verify every commit locally, copy no `.git`
   metadata, and never run `git submodule update` or consult a remote. The root
   repository tree and `HEAD` remain the frozen parent identity; other Profiles
   retain Gitlink-only directories.

The original Source identity and revision remain authoritative and are also the
internal workspace base. No synthetic commit is published or substituted.

### 5.3 Backend integration

Local Process, Docker, and Remote Docker will call the same helper. Remote
Docker will compute its sync fingerprint and run exact rsync only after the clean
local snapshot is complete. No backend may fall back to copying the cache
working tree.

The Scripted backend continues to use its explicit fixture behavior, but any
path that accepts a real `RuntimeAttemptContext` must obey the same revision
field validation.

For the CUDA Kernel Profile, the exact private target may additionally declare
one absolute `remote_ccache_seed`. The Remote backend copies that known path
with `cp -a --reflink=auto` into the exact Attempt-owned remote workspace and
excludes only that `.ccache/` copy from later source rsync. The shared seed is
never mounted writable, mutated, listed, or cleaned; the writable copy is
removed with the Attempt workspace. Kernel source loading uses
`setup.py build_ext --inplace` plus the fixed workspace `PYTHONPATH`, avoiding
both writes to the read-only container root and PEP 517 network access.

### 5.4 Failure and cleanup

Materialization failures remain `RuntimeBackendUnavailable` with the public
reason `workspace_prepare_failed`. Private exception text may support local
debugging but must not enter public artifacts.

The workspace ledger transition is:

```text
declared -> created -> released
         \-> create_failed
```

`created` occurs only after extraction, deterministic repository creation, tree
verification, and cleanliness verification succeed. On any earlier failure, the
backend removes only the exact newly declared Attempt workspace, records
`create_failed`, and leaves the source cache unchanged. Existing exact cleanup
and Integrity checks remain authoritative.

## 6. Security and safety properties

- All Git operations read only the configured local source repository and the
  exact frozen revision.
- Archive extraction is descriptor/path guarded and rejects entries capable of
  escaping the Attempt workspace.
- No command enumerates hosts, ports, processes, containers, or candidate
  targets.
- Remote execution continues to address only the supplied private target with
  existing SSH identity verification.
- Target values, raw Runtime handles, local source paths, and private exception
  text remain absent from public artifacts.
- Cleanup targets are exact Attempt-owned paths and handles; no wildcard or
  broad cache cleanup is introduced.
- The optional ccache seed is an exact configured path and is read only through
  a direct copy into an Attempt-owned destination. No command searches for
  caches or falls back to another path.

## 7. Compatibility

- Public Runtime contracts and schemas remain version `v1`.
- RunManifest, Comparability Key, Cohort ID, Attempt ID, historical results, and
  v0.5 artifact hashes do not change solely because of this fix.
- Backend constructor and CLI behavior remain unchanged.
- Internal `RuntimeAttemptContext` construction sites must provide an explicit
  revision; omission is a contract error rather than an implicit `HEAD`
  fallback.
- Resume continues to preserve valid existing Attempts. The failed resumed
  canary remains append-only evidence; the corrected run uses a new retry or new
  output root according to the existing retry contract.
- Historical v0.5 result files and Task manifests remain byte-identical. Replay
  expected outcomes are derived from raw F2P/P2P counts, which corrects three
  inconsistent `#140557` summary labels without rewriting provenance.
- The single v0.5 Gold patch whose context predates the frozen base's
  `enable_gqa` parameter is represented by a replay-only strict patch. A
  checked-in provenance sidecar binds the Task, base commit, original Gold,
  Admission evidence, strict bytes, and context-only normalization method.
  Replay rejects any mismatch and never enables global fuzzy patching.

## 8. Test design

Implementation follows red-green-refactor.

1. Add a backend regression test with a committed source revision plus an
   ignored directory, an untracked file, and a tracked working-tree edit. The
   materialized workspace must contain only the committed revision bytes, omit
   ignored/untracked state, and pass the authoritative cleanliness check. This
   test must fail against the current `copytree` implementation.
2. Add an invalid-revision test requiring `workspace_prepare_failed`, a
   `create_failed` ledger transition, no published lease, no leftover Attempt
   workspace, and unchanged source-cache bytes.
3. Extend Docker/Remote fake-runner tests to prove rsync/container setup consumes
   the clean revision snapshot and adds no discovery operation.
4. Update orchestrator, replay, and conformance tests to prove the exact revision
   is propagated into every `RuntimeAttemptContext`.
5. Add nested-submodule, exact ccache-seed ownership, no-install in-place build,
   historical count precedence, and strict-Gold provenance tests.
6. Run focused backend/orchestrator/replay tests, all Runtime tests, and the full
   repository suite.

## 9. Runtime acceptance after implementation

1. Re-run the exact one-Task Remote CPU scripted canary without any preliminary
   network probe. Require a valid Attempt, Runtime conformance evidence, all
   Integrity checks, and complete resource release.
2. Run representative exact CUDA Overlay and CUDA Kernel Profile canaries,
   serially, only after Remote CPU passes.
3. Run the frozen 17 baseline + 17 gold + 51 legacy final-patch replay against
   the same exact target. Require 85 real results, per-case difference
   attribution, unchanged historical v0.5 hashes, and complete cleanup evidence.
4. Update R-05, R-06, R-07, R-08, and R-10 only when their direct artifacts
   prove the requirements. Do not infer success from target reachability or from
   deterministic fixtures.
5. Re-run the full release audit before changing `V06-RELEASE` from `Blocked`.

## 10. Definition of done

The fix is complete only when the regression test demonstrates that cache
pollution cannot enter an Attempt, all test scopes pass, the exact Remote
CPU/CUDA and replay artifacts satisfy their existing gates, historical hashes
remain unchanged, documentation reflects the new evidence, and the final
release audit has no Must item left outside `Passed`.
