# OpBench v0.6 M3 Canonical Action Service / Adapter / MCP Plan

> **Status:** Complete (2026-07-17)
>
> **Milestone scope:** Acceptance A-01 through A-12. All fixtures are local and
> deterministic; no network probes or remote runtime operations are part of M3.

## Goal

Provide one authoritative, versioned action execution path for every Agent-facing
operation. CLI and MCP are transports over the same service, adapters receive only
the public task launch input and an action client, and the v0.5 Action Bridge remains
available during migration.

## Design decisions

- `ActionRequest` and `ActionObservation` remain the only wire contracts.
- The service owns session/action id validation, capability checks, idempotency,
  deadlines, action/test/command/output accounting, and stable error mapping.
- Workspace mutations and diffs delegate to the M2 `AuthoritativeWorkspace`.
- `command_run` accepts argv only. Policy prefixes are parsed as argv and must match
  complete leading arguments; shell strings and absolute/traversing cwd values fail
  closed.
- `test_run` resolves only a control-plane registry entry. Agent-supplied command
  text is never executed as a test selector.
- Command execution is behind a runtime-owned backend receiving a canonical relative
  cwd, so the Agent Adapter never receives a host path or Workspace object.
- `session_finish` freezes once; repeated same requests reuse the exact observation,
  and later finish IDs reuse the same immutable patch result without a second side
  effect.
- CLI and MCP validate/construct the same request and call the same service method.
- The reference Adapter channel is an in-process, JSON-only API/data-minimization
  boundary. It is not a security sandbox for adversarial Python code; an untrusted
  Adapter must run behind process/IPC isolation while preserving the same client
  contract. Process isolation and the real-Codex canary are M6 conformance work.
- Action exchanges provide the M3 request/observation audit evidence. M4 will bind
  those exchanges into the append-only, hash-chained lifecycle `EventRecord` stream.
- M3 validates the standard Adapter boundary with deterministic local fixtures and
  keeps the v0.5 bridge green. Migrating the real Codex launcher to that boundary
  and executing the real canary remain explicit M6 conformance gates.
- Backend output currently fails safe by replacing an entire output field when the
  public-artifact scanner finds a sensitive value. M4/M5 may improve diagnostics by
  normalizing known workspace paths and applying targeted redaction without
  weakening the public/private boundary.
