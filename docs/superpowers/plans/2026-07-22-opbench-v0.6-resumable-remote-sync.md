# OPBench v0.6 Resumable Remote Sync Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Allow a large exact remote source sync to continue within the same Runtime attempt after transient transport interruption.

**Architecture:** A private backend helper owns a bounded three-invocation loop around the existing exact `rsync` command. Both initial preparation and later changed-workspace synchronization call it, while the existing outer cleanup path remains authoritative after permanent failure.

**Tech Stack:** Python 3.12, `unittest`, OPBench v0.6 Runtime backend contracts, rsync over the configured SSH transport.

## Global Constraints

- Reuse only the configured target and exact attempt-owned remote workspace.
- Do not add discovery, network probes, alternate targets, or broad cleanup.
- Preserve `remote_source_sync_failed` after the bounded attempts are exhausted.
- Do not change manifest, Runtime Profile, source fingerprint, resource identity, or evaluation semantics.

**Interfaces:**

- [ ] **Step 3: Implement the minimal helper**

Add a module constant of three attempts, include `--partial` in
`_rsync_command`, and add this behavior:

```python
def _sync_remote_workspace(...):
    command = _rsync_command(...)
    for _ in range(_REMOTE_SYNC_MAX_ATTEMPTS):
        result = argv_runner(command, None, timeout_ms)
        if result.exit_code == 0 and not result.timed_out:
            return
    raise RuntimeBackendUnavailable("remote_source_sync_failed")
```

Replace the duplicate one-shot initial and changed-workspace rsync blocks with
the helper.
