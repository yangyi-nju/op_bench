# OpBench v0.6 M1 Contracts and Run Manifest Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the strict, versioned, deterministic contract and run-manifest foundation required by OpBench v0.6 while preserving v0.5 dataset compatibility.

**Architecture:** Add a zero-dependency `op_bench.runtime` package. Immutable Python contracts perform fail-closed parsing, canonical JSON provides content identity, a run-manifest builder freezes comparability and attempt matrices, and a legacy adapter projects the existing v0.5 dataset into the new contracts. JSON Schema remains a separately testable wire-format artifact.

**Tech Stack:** Python 3.9+ standard library, frozen dataclasses, JSON Schema Draft 2020-12 artifacts, `unittest`, existing OpBench task/dataset/registry modules.

## Global Constraints

- M1 development and verification is local and offline. Later milestones may use the explicitly configured OpenAI and remote-runtime paths required by their acceptance tests, but must not perform network probes, port scans, public-IP/cloud-metadata discovery, identity enumeration, or checks outside the current Attempt's declared resources.
- Preserve the v0.5 default CLI and data model; v0.6 contracts are additive until the later runtime migration.
- Reject unknown schema versions, unknown fields, missing identities, booleans used as integers, non-finite numbers, and values outside declared enums.
- Canonical hashes use UTF-8 JSON with sorted keys, compact separators, `ensure_ascii=False`, and a `sha256:` prefix.
- Every wire-level contract includes both `contract_type` and `schema_version`.
- M1 is complete only when Acceptance Matrix C-01 through C-08, full unit tests, v0.5 dataset validation, and an offline example-manifest rebuild pass.
